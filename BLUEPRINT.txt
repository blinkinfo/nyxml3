SECTION 1 — WHAT WAS TRIED AND WHY IT FAILED
================================================================================

The following approaches were tested and rejected before arriving at this model.
Your agent must NOT repeat these dead ends.

1.1  Pure U/D candle direction patterns (3-4 char and 3-9 char)
     - Encoded each closed candle as U (close > open) or D (close <= open)
     - Mined all pattern combinations of length 3 to 9
     - Applied conflict resolution: longest pattern wins when multiple fire
     - Result: all 16 x 3-4 char patterns clustered within +/-5.4% of 50% WR
     - Train WR: 55.8%, Test WR: 50.3% — classic overfit, no real edge
     - Conclusion: raw U/D direction sequences carry almost zero predictive
       signal on 5-min BTC. Do not use.

1.2  LightGBM with RSI, MACD, and standard indicators
     - User confirmed this was tried previously
     - RSI and MACD are derived from the same price data, lag too much on 5-min
     - Conclusion: lagging indicators do not work on this timeframe. Do not use.

1.3  LightGBM with direction lags only (dir_1..dir_4)
     - Feature importance: dir_1..4 ranked near zero
     - Direction lags added no signal on top of structural features
     - Conclusion: do not include direction lag features.

1.4  Three-layer ensemble (patterns + structure score + LightGBM)
     - Combined pattern WR score, candle structure score (0-3), and LightGBM
     - Pattern layer anchored the combined score near 0.50, dragging WR down
     - Best test WR: 55.56% at 7.18 trades/day — not sufficient
     - Conclusion: removing the pattern layer was the correct move.

1.5  LightGBM with structural + multi-TF + funding (no CVD)
     - Added 15m/1h body ratios, EMA9 slope, funding rate, funding zscore
     - Best test WR: ~56% at ~20 trades/day — still below target
     - Conclusion: CVD was the missing signal needed to break the ceiling.


SECTION 2 — WHAT ACTUALLY WORKS: THE WINNING MODEL
================================================================================

Model type    : LightGBM binary classifier (single model, no ensemble layers)
Signal source : Candle structure + multi-timeframe confluence + CVD
No patterns   : U/D direction sequences are NOT used anywhere
No indicators : No RSI, MACD, Bollinger Bands, or any standard TA indicator

The edge comes from three sources working together:
  (a) 1h and 15m candle body size — when the higher timeframe candle is strongly
      directional, the 5-min continuation probability rises meaningfully
  (b) 5-min candle structure — body size and wick ratios on recent candles
      show conviction vs indecision
  (c) CVD (buy/sell volume balance) — who is actually winning the volume battle
      inside each candle, normalized and rolled over 5 and 20 candles

Top feature importances (gain, from final trained model):
  1.  body_ratio_1h       — strongest single predictor

3.5  CVD (Cumulative Volume Delta) — 5-min buy/sell volume
     Source    : MEXC futures kline REST API (NOT ccxt — direct HTTP)
     Endpoint  : https://contract.mexc.com/api/v1/contract/kline/BTC_USDT
     Symbol    : BTC_USDT (underscore, not slash — this is the futures REST format)
     Interval  : Min5
     Params    : interval=Min5, start=<unix_seconds>, end=<unix_seconds>
     Response  : JSON with data.time[], data.open[], data.high[], data.low[],
                 data.close[], data.vol[]
     File      : btcusdt_5m_cvd.csv
     Columns   : timestamp, open, high, low, close, volume, buy_vol, sell_vol

     IMPORTANT — CVD CALCULATION METHOD:
     The MEXC futures kline does NOT return native buy/sell volume split.
     We approximate it using the candle close position (standard method):
       if (high - low) > 0:
         buy_vol  = volume * (close - low)  / (high - low)
         sell_vol = volume * (high - close) / (high - low)
       else:  (doji candle, high == low)
         buy_vol  = volume * 0.5
         sell_vol = volume * 0.5

     This is a proxy, not real taker data. It is noisier than true CVD but
     still adds meaningful signal (ranked 6th in feature importance).
     If real taker buy/sell data becomes available, replace this proxy.

     Pagination: fetch 100 candles per request (100 * 5min = 500 min window).
     Timestamps from API are Unix seconds. Convert to UTC datetime after fetch.
     Use httpx (or requests) directly — do not use ccxt for this endpoint.
     Rate limit: safe at 0.05s sleep between requests (~20 req/s).


SECTION 4 — INDEX ALIGNMENT (CRITICAL — READ CAREFULLY)
================================================================================

This is the most important section. Getting this wrong causes lookahead bias
and produces fake backtest results that collapse in live trading.

Timeline at prediction time:
  - Candle N   : LIVE, currently forming. NEVER use any data from candle N.
  - Candle N-1 : Most recently CLOSED candle. This is the freshest data you can use.
  - Candle N-2 : Second most recent closed candle.
  - Candle N-3 : Third most recent closed candle.
  - Candle N+1 : The NEXT candle that will form. This is what we PREDICT.

In code (pandas DataFrame, 0-indexed rows, sorted ascending by time):
  At DataFrame row index i:
    - Candle N   = row i      (current live candle — DO NOT USE)
    - Candle N-1 = row i-1    (df.shift(1))
    - Candle N-2 = row i-2    (df.shift(2))
    - Candle N-3 = row i-3    (df.shift(3))
    - Candle N+1 = row i+1    (prediction target)

  TARGET construction:
    target[i] = 1 if close[i+1] > close[i] else 0
    Note: close[i] is candle N — used ONLY to define the direction of N+1.
    It is NOT used as a feature. Using it only as a reference for the
    target label is acceptable and does not constitute lookahead.

  FEATURE construction rule: every .shift(k) call must have k >= 1.
  Any feature using .shift(0) or no shift uses candle N data — FORBIDDEN.

  For multi-timeframe merges:
    The lookup timestamp is df5['timestamp'].shift(1) — the N-1 candle timestamp.
    Use pd.merge_asof(..., direction='backward') to find the most recent
    15m/1h/funding candle whose timestamp <= N-1 timestamp.
    This guarantees no future data bleeds into features.


SECTION 5 — FEATURE ENGINEERING (EXACT FORMULAS)
================================================================================

All features are computed from candle N-1 and earlier. All ATR values use
ATR14 (14-period exponential moving average of True Range).

ATR14 formula:
  true_range = max(high - low, abs(high - prev_close), abs(low - prev_close))
  ATR14 = EWM of true_range with alpha=1/14, min_periods=14, adjust=False

--- 5m STRUCTURAL FEATURES (from btcusdt_5m_5mo.csv) ---

body_ratio_n1 = (close[i-1] - open[i-1]) / ATR14[i-1]
  Signed: positive = bullish candle, negative = bearish candle
  Magnitude = conviction strength relative to recent volatility

body_ratio_n2 = (close[i-2] - open[i-2]) / ATR14[i-2]
body_ratio_n3 = (close[i-3] - open[i-3]) / ATR14[i-3]

upper_wick_n1 = (high[i-1] - max(open[i-1], close[i-1])) / ATR14[i-1]
  Always >= 0. Large value = price rejected upward move (bearish signal)

upper_wick_n2 = (high[i-2] - max(open[i-2], close[i-2])) / ATR14[i-2]

lower_wick_n1 = (min(open[i-1], close[i-1]) - low[i-1]) / ATR14[i-1]
  Always >= 0. Large value = price rejected downward move (bullish signal)

lower_wick_n2 = (min(open[i-2], close[i-2]) - low[i-2]) / ATR14[i-2]

volume_ratio_n1 = volume[i-1] / rolling_mean(volume, 20)[i-2]
  Note: rolling mean uses data up to i-2 (not i-1) to avoid any lookahead.
  In code: vol_mean = df['volume'].shift(1).rolling(20).mean()
           volume_ratio_n1 = df['volume'].shift(1) / vol_mean

volume_ratio_n2 = volume[i-2] / rolling_mean(volume, 20)[i-3]

--- 15m MULTI-TIMEFRAME FEATURES (from btcusdt_15m_5mo.csv) ---

Merge: find the latest 15m candle with timestamp <= 5m candle N-1 timestamp.
Use pd.merge_asof(left=df5[ts_n1], right=df15, direction='backward').

body_ratio_15m = (close_15m - open_15m) / ATR14_15m
  ATR14 computed on the 15m DataFrame independently.
  Signed same as body_ratio_n1.

dir_15m = sign(close_15m - open_15m)
  Values: +1 (up), -1 (down), 0 (doji)

volume_ratio_15m = volume_15m / rolling_mean(volume_15m, 20)
  Rolling mean computed on 15m DataFrame.

--- 1h MULTI-TIMEFRAME FEATURES (from btcusdt_1h_5mo.csv) ---

Merge: find the latest 1h candle with timestamp <= 5m candle N-1 timestamp.

body_ratio_1h = (close_1h - open_1h) / ATR14_1h
  This is the single strongest feature in the model.

dir_1h = sign(close_1h - open_1h)

ema9_slope_1h = (EMA9_1h[t] - EMA9_1h[t-1]) / ATR14_1h[t]
  EMA9 computed on 1h close with span=9, adjust=False.
  Normalized by ATR14 to make it scale-invariant.
  Positive = uptrend accelerating, Negative = downtrend accelerating.

--- FUNDING RATE FEATURES (from btcusdt_funding_5mo.csv) ---

Merge: find the latest funding record with timestamp <= 5m candle N-1 timestamp.
MEXC funding rate is every 8 hours. There will be many 5m candles sharing
the same funding value — this is correct and expected.

funding_rate = raw funding rate value (e.g., 0.0001 = 0.01%)
  Positive = longs pay shorts (market is long-biased, mean reversion risk)
  Negative = shorts pay longs (market is short-biased)

funding_zscore = (funding_rate - rolling_mean(funding_rate, 24)) 
                 / rolling_std(funding_rate, 24)
  Window: 24 funding periods = 24 * 8h = 8 days of history
  Extreme positive zscore = crowded long = potential DOWN signal
  Extreme negative zscore = crowded short = potential UP signal
  If std == 0, set zscore = NaN (will be dropped in dropna step)

--- CVD FEATURES (from btcusdt_5m_cvd.csv) ---

Merge on timestamp (exact match — CVD file covers same 5m candles as main file).
Use N-1 CVD data only (shift by 1 in merge or shift columns after merge).

delta[i]       = buy_vol[i] - sell_vol[i]
  Positive = buyers won this candle, Negative = sellers won

delta_ratio[i] = buy_vol[i] / (buy_vol[i] + sell_vol[i])
  Range: 0.0 to 1.0. Values above 0.5 = more buying than selling.
  Clip denominator minimum to avoid division by zero.

cvd_delta[i]   = delta[i] / ATR14_cvd[i]
  Single-candle delta normalized by 5m ATR14.

cvd_5[i]       = sum(delta[i-4] to delta[i]) / ATR14_cvd[i]
  Rolling 5-candle cumulative delta, normalized. Short-term buy/sell pressure.

cvd_20[i]      = sum(delta[i-19] to delta[i]) / ATR14_cvd[i]
  Rolling 20-candle cumulative delta, normalized. Medium-term buy/sell pressure.

cvd_trend[i]   = cvd_5[i] - cvd_20[i]
  Momentum of CVD: positive = recent buying accelerating vs recent average.

All CVD features use shift(1) when merging to ensure only N-1 data is used.

--- TIME FEATURES ---

These were computed but ranked low in importance. Include them:

hour_sin = sin(2 * pi * hour / 24)   where hour = UTC hour of N-1 candle
hour_cos = cos(2 * pi * hour / 24)
day_of_week = day of week integer (0=Monday, 6=Sunday) of N-1 candle

SECTION 9 — THRESHOLD SELECTION
================================================================================

The model outputs a probability between 0.0 and 1.0 for each candle.
We only place a trade when probability >= threshold.
Higher threshold = fewer trades, higher win rate.
Lower threshold = more trades, lower win rate.

Threshold sweep was done on the VALIDATION set (not test set).
Sweep range: 0.50 to 0.80 in steps of 0.005.
Selection criterion: WR >= 59% with maximum trades/day.

SECTION 11 — LIVE BOT IMPLEMENTATION
================================================================================

11.1  WHAT TO BUILD

The live bot needs to do the following on every 5-min candle close:

  Step 1 : Detect that candle N-1 has just closed (5-min boundary)
  Step 2 : Fetch the latest OHLCV data for all timeframes
  Step 3 : Compute all 22 features for the current row
  Step 4 : Run model.predict([feature_row]) to get probability
  Step 5 : If probability >= 0.590, place a binary options trade
           Direction: UP if predicted class == 1, DOWN if class == 0
           Expiry: next 5-min candle (candle N+1)
  Step 6 : Log the trade, wait for the next candle close

11.2  DATA FETCHING IN LIVE MODE

On each candle close, fetch the minimum data needed:

  5m OHLCV   : Last 50 candles (need 20 for volume rolling mean + ATR14 warmup)
               exchange.fetch_ohlcv('BTC/USDT', '5m', limit=50)
               Use the last candle as N-1 (most recently closed).

  15m OHLCV  : Last 30 candles
               exchange.fetch_ohlcv('BTC/USDT:USDT', '15m', limit=30)
               Find the candle whose timestamp <= timestamp of 5m N-1 candle.

  1h OHLCV   : Last 20 candles
               exchange.fetch_ohlcv('BTC/USDT:USDT', '1h', limit=20)
               Find the candle whose timestamp <= timestamp of 5m N-1 candle.

  Funding    : Latest funding rate
               exchange.fetch_funding_rate('BTC/USDT:USDT')
               Returns current funding rate directly (no history needed live).
               For zscore: maintain a rolling buffer of last 24 funding values.

  CVD        : Fetch last 25 candles from MEXC futures kline endpoint
               GET https://contract.mexc.com/api/v1/contract/kline/BTC_USDT
               Params: interval=Min5, start=<now-125min in seconds>, end=<now in seconds>
               Compute buy_vol/sell_vol proxy from OHLCV, then compute CVD features.

11.3  FEATURE COMPUTATION IN LIVE MODE

  All feature formulas are identical to the training formulas in Section 5.
  The only difference is you work with arrays of recent candles instead of
  a full DataFrame.

  Key rule: the "current" row you compute features for is row i.
  Row i represents the prediction moment (live candle N is forming).
  All features look at index i-1 (N-1) and earlier.
  You feed one feature row [22 values] into the model.

  Practical implementation:
    - Keep a rolling buffer (deque) of the last 50 five-min candles
    - Recompute ATR14 using the last 14 candles in the buffer
    - Feature row is computed from buffer[-1] (N-1) and buffer[-2] (N-2), etc.
    - Same for 15m (buffer of last 25), 1h (buffer of last 15)

11.4  MODEL FILE

  After training, save the model with:
    model.save_model('btc_ensemble_v3.lgb')

  Load in live bot with:
    import lightgbm as lgb
    model = lgb.Booster(model_file='btc_ensemble_v3.lgb')
    prob = model.predict(feature_array)[0]   # single row

  The model file is fully portable. Copy it to your live bot environment.
  It does not require retraining — use the saved weights directly.

11.5  TIMING

  Do NOT run the bot on a fixed clock interval (e.g. every 5 minutes).
  5-min candles close at :00, :05, :10, :15 ... :55 past each hour UTC.
  Run the prediction logic immediately AFTER candle N-1 closes.

  Recommended approach:
    - Subscribe to MEXC WebSocket kline stream for BTC/USDT 5m
    - On receiving a candle with x.T (candle close time) in the past, trigger
    - Or: poll every 10 seconds, check if a new closed candle has appeared

  Allow ~2-5 seconds for data propagation before fetching features.
  Binary options platforms typically require the trade to be placed before
  the next candle opens — you have a 5-minute window.

11.6  RETRAIN SCHEDULE

  The model should be retrained periodically to adapt to market regime changes.
  Recommended: retrain monthly using rolling 5-month window.
  Keep the threshold tuning step in every retrain — do not hardcode 0.590
  forever. Markets change and the optimal threshold may shift.

  Retrain checklist:
    1. Fetch latest 5 months of all 5 data files
    2. Rerun full feature engineering pipeline
    3. Retrain LightGBM with same params (early stopping will find best iter)
    4. Re-run threshold sweep on new validation set
    5. Select threshold that gives WR >= 59% at maximum trades/day
    6. Save new model file and deploy
    7. Keep old model file as backup for 1 week

SECTION 14 — CRITICAL RULES — DO NOT VIOLATE
================================================================================

1. NEVER use candle N (the live/forming candle) in any feature.
   All features must use .shift(1) or greater.

2. NEVER shuffle the data before splitting.
